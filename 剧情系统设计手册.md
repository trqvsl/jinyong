# 剧情系统设计手册

> 金庸群侠传的灵魂是"**选择有重量**"——你在牛家村救不救丘处机，会改变全真教对你的态度、
> 你的侠义名望、日后能否学得全真剑法、乃至某位 NPC 是否愿意加入你。
> 本手册把剧情系统设计成一个**随选择演化的世界**：所有剧情读写同一份"世界状态"，
> 节点之间通过它彼此因果。这是与战斗系统平级、需要独立打磨的核心系统。

---

## 一、核心理念：选择塑造世界

剧情系统的本质不是"可跳转的节点图"，而是"**一个被玩家选择不断改写的江湖**"。

玩家的一次选择，影响沿四个层次传播（这是本系统的脊柱）：

| 影响层次        | 例子                     | 落到哪                                  |
| ----------- | ---------------------- | ------------------------------------ |
| ① 自身收益/属性   | 得银两、习武功、声望涨            | `player`（属性/道具/武功）                   |
| ② NPC 好感与关系 | 丘处机好感+15               | `player.relations`（已存在）              |
| ③ NPC 命运/阵营 | 招募段誉入队、杨康黑化、全真教敌视      | `world.npcs` / `world.factions`      |
| ④ 后续剧情/其他事件 | 侠义够高→洪七公主动传功；未救人→日后被冷遇 | `condition` 查询 + `world.arcs` + 涌现事件 |

四层统一为同一件事：**对世界状态的声明式写入（consequence）与查询（condition）**。玩家不必记得自己做过什么，但**世界记得**。

## 二、架构分层（依赖方向，单向不可逆）

```
data/story/            ← 剧情内容（纯数据，只有"发生了什么"）
  schema.ts              数据模型类型
  volumes/shendiao.ts    射雕卷（纯数据节点）
  volumes/common.ts      通用奇遇
  world-events.ts        涌现事件（由世界状态触发）
  index.ts               聚合 + 查询

game/story/             ← 剧情引擎（只有"怎么发生"，不认识任何具体剧情）
  state.ts               WorldState 定义 / 初始化 / 迁移
  consequences.ts        声明式后果解释器（applyConsequences）
  conditions.ts          条件解释器（checkCondition）
  engine.ts              节点结算 / 流转 / 战斗衔接
  world-scheduler.ts     涌现事件扫描

screens/EventScreen.tsx ← 纯展示，调引擎
App.tsx                 ← 只做路由编排
```

**铁律**：`game/story/*` 永远不 import `data/story/volumes/*`，不认识任何剧情 id / NPC id。引擎只认 `schema.ts` 的类型。

## 三、世界状态 WorldState（核心）

### 3.1 与 Player 的关系

`Player` 保持现有结构（战斗属性 + `reputation`/`alignment`/`relations`/`gold`）。新增 `WorldState` 承载"江湖记忆"，挂在 `player.world` 上随存档持久化（旧存档迁移补默认空状态）。

- `reputation` / `relations` / `gold` 已在 `player` 上，**就地复用**，不搬迁（避免大改）。
- `world.npcs` / `world.factions` / `world.arcs` / `world.flags` 是**新增**的剧情世界维度。
- 引擎统一寻址——**作者只写"改什么"，引擎知道存哪**，不必关心字段在 player 还是 world。

### 3.2 善恶值 karma（新增数值源头）

金庸群侠传需要数值化的善恶路线，而当前 `player.alignment` 只是 `"正"|"邪"|"中"` 三值枚举。引入：

```ts
player.karma: number   // -100(邪) ~ +100(正)，0 为中性
```

`alignment` 枚举**由 karma 派生**（karma ≥ 30 → "正"；≤ -30 → "邪"；否则 "中"），保留枚举接口供 UI/旧逻辑使用。

> **同步约定（重要）**：`player.alignment` 是派生字段，**不得直接用 consequence 修改**。引擎在 `applyConsequences` 中凡涉及 `karma` 变更，结算后自动 `player.alignment = deriveAlignment(player.karma)` 写回，杜绝两者不一致。

> 决策点（待你拍板）：是否允许剧情用 `{kind:"karma", set:-50}` 之类**强制**设善恶（如被邪功诱惑直接入邪）？我建议允许（见 3.4 的 set），这正是金庸式"一念成魔"的支点。

### 3.3 WorldState 结构

```ts
interface WorldState {
  version: number                                  // 结构版本号，供 migrateWorld 升级旧存档

  // NPC 命运：好感之外的"关系状态"（好感值仍用 player.relations，不重复存）
  npcs: Record<string, {
    alive: boolean                 // 生死（守门：已死 NPC 不再出现）
    recruited: boolean             // 是否已加入主角队伍（经典招募机制）
    faction: string                // 当前所属阵营（可随剧情转变，如杨康叛全真投金）
    fateTags: string[]             // 命运标记（如 "杨康已黑化"、"段誉已结拜"）
  }>

  // 阵营/门派：对玩家的态度 + 势力消长
  factions: Record<string, {
    attitude: number               // 对玩家的态度 -100~100
    power: number                  // 势力强弱（可用于世界事件/结局）
  }>

  // 主线/事件链进度
  arcs: Record<string, {
    beats: Record<string, "won" | "lost" | "skipped" | "done">
  }>

  flags: Record<string, boolean | number | string>  // 琐碎标记兜底（命名规范见 12.2）

  triggeredEvents: string[]        // 已触发过的涌现事件 id（防重复）
  seenNodes: string[]              // 已进入并结算过 onEnter 的节点 id（防重复结算，见 10.2）
}
```

> 字段**渐进生长**：`npcs`/`factions`/`arcs` 都是 `Record`，写新剧情时随用随加 key，无需全局重构。

### 3.4 默认值约定（查询时绕不开）

`checkCondition` 查某 NPC/faction 时，它可能还没在 `world` 里建条目（玩家首次接触）。**引擎必须按以下默认值处理 missing key**，否则行为不确定：

| 查询对象 | world 无条目时的默认值 | 初始值的来源 |
|---|---|---|
| `npc.alive` | `true` | — |
| `npc.recruited` | `false` | — |
| `npc.faction` | 该 NPC 的**静态定义**（`npcs.ts`） | 静态定义是"底色"，`world.npcs[id].faction` 只在剧情改变它后才覆盖 |
| `npc.fateTags` | `[]` | — |
| `faction.attitude` | `0`（中立） | — |
| `faction.power` | 该阵营的**静态定义**（或 `0`） | 同上，静态是底色、world 存覆盖 |
| `arc.beat` | 未完成 | — |

**原则**：`world` 只存"相对初始状态发生了的变化"，不冗余复制全部 NPC/阵营的初始值。首次写入某 NPC 条目时，缺省字段按上表填。

## 四、因果机制：Consequence（声明式写入）

一次选择/一个战斗结局对世界产生的影响，全部声明式描述。引擎的 `applyConsequences` 认识以下种类：

```ts
// 数值类后果：{ delta } 增减 或 { set } 设为绝对值，二选一。set 用于"强制"场景（如一念入邪 set karma=-50）。
type Consequence =
  // ① 自身收益/属性
  | { kind: "reputation"; delta?: number; set?: number }
  | { kind: "karma"; delta?: number; set?: number }
  | { kind: "gold"; delta?: number; set?: number }
  | { kind: "exp"; delta?: number }
  | { kind: "hp" | "mp"; delta?: number }
  | { kind: "aptitude" | "attack" | "speed"; delta?: number }
  | { kind: "item"; id: string; count?: number }   // 默认 1，负数=扣除
  | { kind: "skill"; id: string }                  // 习得武功（去重）
  // ② NPC 好感（写入 player.relations）
  | { kind: "relation"; npcId: string; delta: number }
  // ③ NPC 命运/阵营
  | { kind: "npcAlive"; npcId: string; alive: boolean }
  | { kind: "npcRecruit"; npcId: string; recruited: boolean }
  | { kind: "npcFaction"; npcId: string; faction: string }
  | { kind: "npcTag"; npcId: string; tag: string; add?: boolean }   // 默认 add=true，false=移除
  | { kind: "factionAttitude"; factionId: string; delta?: number; set?: number }
  | { kind: "factionPower"; factionId: string; delta?: number; set?: number }
  // ④ 主线进度 + 标记
  | { kind: "arcBeat"; arcId: string; beat: string; result: "won" | "lost" | "skipped" | "done" }
  | { kind: "flag"; name: string; value: boolean | number | string }
```

> **clamp 约定**：`applyConsequences` 对有界字段做夹断——`karma`/`factionAttitude`/`relation` 夹到 `[-100,100]`，`hp`/`mp` 夹到 `[0, max]`。（`relation` 的夹断沿用 `relations.ts` 已有逻辑，保持一致。）

> **边界**：Consequence 种类是有限枚举。遇到新需求，优先用 `flag`/`npcTag` + condition 表达；只有高频通用的变更才升级为新 kind。不为单一剧情加专属 consequence。

## 五、查询机制：Condition（声明式读取/分支）

节点、选项、事件、涌现触发都可挂条件，控制可见/可选/可触发：

```ts
type Condition =
  | { kind: "reputation"; gte?: number; lte?: number }
  | { kind: "karma"; gte?: number; lte?: number }              // 善恶区间（正线/邪线分支）
  | { kind: "relation"; npcId: string; gte?: number; lte?: number }  // 好感上下限
  | { kind: "npcAlive"; npcId: string; alive?: boolean }
  | { kind: "npcRecruited"; npcId: string }
  | { kind: "npcHasTag"; npcId: string; tag: string }
  | { kind: "factionAttitude"; factionId: string; gte?: number; lte?: number }
  | { kind: "arcBeat"; arcId: string; beat: string; result?: "won" | "lost" | "skipped" | "done" }
                                                                // result 缺省 = 该 beat 已完成（不论结果）
  | { kind: "flag"; name: string; eq?: boolean | number | string }
  | { kind: "hasItem"; id: string }
  | { kind: "hasSkill"; id: string }
  | { kind: "and"; items: Condition[] }
  | { kind: "or"; items: Condition[] }
  | { kind: "not"; item: Condition }
```

> `flag` 仅支持等值（`eq`）。若需数值区间判断，应改用结构化字段（`karma`/`relation`/`reputation` 等），不要把数值塞进 flag。

## 六、涌现事件（"其他事件"被世界变化触发）

除玩家主动触发的剧情外，有一类**世界事件**：当世界状态满足条件时自动浮现，体现"世界记得你的选择"。

```ts
interface WorldEvent {
  id: string
  trigger: Condition           // 满足则可触发（如 karma ≥ 30 且未触发过）
  once: boolean                // 通常 true，触发后记入 triggeredEvents
  event: StoryEvent            // 触发后进入的剧情事件
}
```

示例——侠义之名远播，洪七公主动现身：
```ts
{
  id: "hongqi-gong-seeks-you",
  trigger: { kind: "and", items: [
    { kind: "karma", gte: 30 },
    { kind: "relation", npcId: "hongqi", gte: 0 },   // 且与洪七公无恶感
  ]},
  once: true,
  event: { /* 洪七公传降龙十八掌的剧情事件 */ },
}
```

**触发时机与防循环**（MVP）：每次回到主菜单时，`world-scheduler` 扫描 `WorldEvent`，触发**第一个**满足 `trigger` 且未在 `triggeredEvents` 中的，弹给玩家；**每轮只触发一个，且不递归扫描**（事件内的 consequence 改了世界状态也不会在本轮连锁触发），避免循环。未来可扩展为按天数/地点触发。

## 七、节点 / 选项 / 事件 / 流转

### 7.1 流转 Transition

```ts
type Transition =
  | { type: "end" }                                                      // 回主菜单
  | { type: "goto"; nodeId: string }                                     // 同事件下一节点
  | { type: "branch"; cases: { when: Condition; then: Transition }[]; else?: Transition }  // 条件分叉
  | { type: "battle"; enemyId?: string; useLocationPool?: boolean;
      lethal?: boolean;                                                  // true=战败即结局（见 7.4）
      onWin?: Outcome; onLose?: Outcome; onFlee?: Outcome }
  | { type: "gotoEvent"; eventId: string }                               // 跨事件串联
  | { type: "gameOver"; endingId?: string }                              // 死亡/结局（接结局系统）
```

`branch` 让"因果"从静态读写升级为**动态选路**：根据世界状态自动走不同分支，无需玩家选择。

### 7.2 战斗结局 Outcome

```ts
interface Outcome {
  text: string
  consequences?: Consequence[]
  then?: Transition                     // 战后继续流转，默认 end（"打赢接下一段"靠它）
}
```

- `onWin`/`onLose`/`onFlee` 任一缺省时，该结局默认 `then: { type: "end" }`。
- 战斗的胜负由 `BattleScreen` 决定；引擎只负责按结果取对应 `Outcome` 并结算。

### 7.3 节点形态

```ts
interface Choice {
  id: string
  text: string
  description?: string
  condition?: Condition                 // 不满足则隐藏
  consequences?: Consequence[]          // 选了立即结算（影响玩家+世界）
  transition: Transition
  resultText?: string
}

interface StoryNode {
  id: string
  title?: string
  text: string
  speaker?: string
  onEnter?: Consequence[]               // 进入节点时结算（幂等，见 10.2）
  choices?: Choice[]                    // 有 = 选择节点
  autoNext?: Transition                 // 无 choice 时展示后自动流转（可用 branch 条件分流）
}
// 约定：choices 与 autoNext 至少有其一；都无 = 终点节点（等价 end）
```

四种节点形态都能表达：
- **选择节点**：`choices`（每个 choice 有自己的 transition）
- **纯叙事节点**：`autoNext`（播完文字自动进下一节点）
- **条件分叉节点**：`autoNext: { type: "branch", ... }`（按世界状态自动分流）
- **终点节点**：无 choices、无 autoNext

### 7.4 战败与死亡语义

- **默认战败**：剧情战斗 `onLose` = 重伤脱战继续（沿用 BattleScreen 的"回 30% 气血"），不是 Game Over。绝大多数战斗走这条。
- **致命战斗**：`battle.lethal: true` 时，战败触发 `gameOver`（玩家死亡/坏结局）。用于 Boss 决战、绝境等"输了就结束"的场合。
- `gameOver` transition 预留 `endingId`，未来接多结局系统（与"十四天书"通关目标呼应）。

### 7.5 事件

```ts
interface StoryEvent {
  id: string
  entryNode: string
  nodes: Record<string, StoryNode>
  locationId?: string
  weight?: number
  condition?: Condition
}
```

## 八、因果链示例（"选择有重量"如何落地）

### 8.1 写入：牛家村救丘处机

```ts
// 牛家村·力战（Choice.transition = battle）
transition: {
  type: "battle", enemyId: "guanjun",
  onWin: {
    text: "丘处机长揖：\"阁下仗义出手，丘某佩服。\"传你两式全真剑法要诀，赠碎银。",
    consequences: [
      { kind: "karma", delta: 5 },                          // 侠义↑
      { kind: "reputation", delta: 5 },                     // 声望↑
      { kind: "relation", npcId: "qiuchuji", delta: 15 },   // 丘处机好感↑
      { kind: "factionAttitude", factionId: "quanzhen", delta: 10 }, // 全真教态度↑
      { kind: "item", id: "small-hp-pill", count: 2 },
      { kind: "arcBeat", arcId: "shendiao", beat: "niujia", result: "won" },
    ],
    then: { type: "end" },
  },
  onLose: { text: "你力战不敌，身负重伤……丘处机留药而去。", then: { type: "end" } },
  onFlee: { text: "你虚晃几招脱身而去。", then: { type: "end" } },
}
```

### 8.2 读取：同一选择在后续的回响

这些写入会在**后续任何剧情**里被 condition 读到，形成因果链：
- **条件分叉**（`branch`）：路过重阳宫，进节点后自动分流——
  ```ts
  autoNext: { type: "branch", cases: [
    { when: { kind: "factionAttitude", factionId: "quanzhen", gte: 10 }, then: { type: "goto", nodeId: "quanzhen-teach" } },
  ], else: { type: "goto", nodeId: "quanzhen-reject" } }
  ```
- **涌现事件**：`trigger: { kind: "relation", npcId: "qiuchuji", gte: 50 }` → 丘处机日后主动来寻、传更高深武功。
- **邪线分支**：若牛家村袖手（karma 不涨/反降），后期正派 NPC 选项被 `condition` 隐藏，或 `branch` 把玩家导向邪线节点。

**这就是"每个节点都可能影响其他节点"——通过共享世界状态，无需节点间互相硬编码引用。**

## 九、引擎接口

```ts
// game/story/state.ts
createWorld(): WorldState                              // 初始空世界（含 version）
deriveAlignment(karma: number): Alignment              // karma → 正/邪/中
migrateWorld(raw: any): WorldState                     // 旧存档升级（按 version 补字段）

// game/story/consequences.ts
// 结算：应用 cs → clamp 有界字段 → 若改了 karma 则重算 alignment 写回
applyConsequences(player: Player, world: WorldState, cs?: Consequence[]): { player: Player; world: WorldState }

// game/story/conditions.ts
// 查询：按 3.4 默认值约定处理 missing key
checkCondition(player: Player, world: WorldState, cond?: Condition): boolean

// game/story/engine.ts
enterNode(player, world, event, nodeId): { node, player, world }      // 进节点（onEnter 幂等结算 + 可见选项过滤）
resolveChoice(player, world, event, nodeId, choiceId): { player, world, resultText?, transition }
// 引擎返回 transition 后，由 App 递归处理 goto/branch/gotoEvent（branch 不含副作用，纯路由）

// game/story/world-scheduler.ts
pollWorldEvent(player, world): { player, world, event?: StoryEvent }  // 每轮至多触发一个，不递归
```

`App.tsx` 只按 `transition.type` 路由（end/goto/branch/battle/gotoEvent/gameOver），并在回主菜单时调 `pollWorldEvent`。**不含任何剧情业务逻辑**。

## 十、一致性与可演进性

### 10.1 守门与去重
- **生死守门**：所有让某 NPC 出场的节点，挂 `condition: { kind: "npcAlive", npcId, alive: true }`，避免"已死之人复现"。
- **涌现去重**：世界事件用 `triggeredEvents` 防重复；一次性剧情用 `arcBeat`/`flag` 标记已完成。

### 10.2 onEnter 幂等与进度持久化
- `onEnter` 的 consequences 可能给道具/声望，**重复进入同一节点不能重复结算**。引擎用 `world.seenNodes` 记录已结算的节点 id，二次进入只重放文字、不重算 `onEnter`。
- 多节点连续剧情的"当前节点"也随存档持久化（记入 `world` 或专门的进度字段），避免玩家中途退出后从头重来、跳过已读内容。

### 10.3 演进
- `WorldState` 带 `version`，结构演进时由 `migrateWorld` 按版本号升级旧存档（对标 `migratePlayer`），字段渐进生长，不必预先定义全部 NPC。

### 10.4 可审视
- 可随时 dump `{ karma, reputation, relations, world }` 一眼看清"当前江湖状态"，便于调试与剧情校验。

## 十一、迁移路线（每阶段可跑可玩）

> 关键：**世界状态的 schema 骨架在阶段 1 就立起来**，即使字段先少。不能等剧情铺开再补，否则 flag 会失控。

**阶段 1｜骨架 + 数据化**
1. 定义 `schema.ts`（WorldState/Consequence/Condition/Transition/Choice/Node/Event，含 `branch`/`autoNext`/`gameOver`）
2. 实现 `state.ts`（含 `migrateWorld`）/`consequences.ts`（含 clamp + karma→alignment）/`conditions.ts`（含默认值约定）/`engine.ts`
3. `Player` 加 `karma`、`world`；旧存档迁移（补默认空世界、karma 由原 alignment 推算）
4. 现有 `STORY_EVENTS` 闭包 → 声明式 consequences；战后收尾由 `onWin` 等驱动，移除 `afterBattle` 补丁
5. **验收**：现有剧情行为不变；WorldState 骨架就位，可 dump 查看

**阶段 2｜节点串联（直接因果）**
- 落地 `goto` / `branch` / `autoNext` + 节点表；App 维护当前节点（持久化）；prerequisite 用 condition/branch 表达
- 单事件升级为多节点连续剧情（选择/纯叙事/条件分叉/终点四种形态）
- **验收**：可做"打赢后自动进入下一段""按世界状态自动分流"

**阶段 3｜世界状态 + 涌现事件（间接因果，"整体剧情"成型）**
- `npcs`/`factions`/`arcs` 读写真正生效；`karma` 驱动正邪分支
- `world-scheduler` 落地，世界事件涌现
- **验收**：选择能跨事件传播、世界记得玩家、善恶路线分化

**阶段 4｜主线 Arc / 名场面事件链 / 结局**
- 聚贤庄、华山论剑、光明顶等大事件链（世界观"远期深化"目标）
- 队友招募系统接入（`npcRecruited` 落地为实际队友）
- `gameOver`/`endingId` 接多结局系统（呼应"十四天书"通关目标）

**阶段 5｜（可选）外部格式 + 编辑器**
- 剧情数据迁 JSON/YAML；评估可视化编辑器与因果依赖图

## 十二、设计原则与边界

### 12.1 核心原则
- **内容零逻辑**：`data/story` 下不出现函数，只描述数据与 consequence。
- **引擎零剧情**：`game/story` 下不出现任何具体剧情/NPC/阵营 id。
- **有限枚举**：Consequence/Condition 种类有界；特殊性用 flag/npcTag + condition 表达。
- **世界状态前置**：schema 骨架先于大量剧情定型；字段渐进生长。
- **不过度设计**：阶段 1–3 即达成"独立、可扩展、选择有重量"的核心价值；DSL、全功能世界模拟、编辑器等留待剧情体量真正需要时。

### 12.2 flag 命名规范
`flags` 是扁平 Record，为防冲突与混乱，命名遵循 **`{arc}.{beat}.{tag}` 或 `{npc}.{action}`** 形式（如 `shendiao.niujia.helped`、`yangkang.betrayed`）。且优先使用结构化字段（`arcBeat`/`npcTag`/`karma`），flag 仅作真正琐碎、无法归类的标记。

## 十三、测试与验证策略

剧情引擎是纯函数，**必须用单测守护**（对标战斗引擎的 `verify-*.ts`）：

1. **解释器单测**（`scripts/verify-story.ts`）：
   - `applyConsequences`：每种 kind 的增减/set、clamp 边界、karma→alignment 同步、item 去重、arcBeat 写入。
   - `checkCondition`：每种 kind、missing key 的默认值、and/or/not 组合。
2. **流转单测**：`branch` 命中正确分支、`autoNext` 纯叙事推进、`battle.onWin/onLose` 衔接。
3. **（可选）剧情一致性静态校验**：扫描全部 `data/story`，报告潜在矛盾——如某 NPC 被 `npcAlive:false` 后，是否仍有节点无条件让其出场；某 `arcBeat` 是否被读取但从未被写入。这是大型剧情铺开后防回归的利器。
