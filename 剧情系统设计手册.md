# 剧情系统设计手册

> 金庸群侠传的灵魂是"**选择有重量**"——你在牛家村救不救丘处机，会改变全真教对你的态度、
> 你的侠义名望、日后能否学得全真剑法、乃至某位 NPC 是否愿意加入你。
> 本手册把剧情系统设计成一个**随选择演化的世界**：所有剧情读写同一份"世界状态"，
> 节点之间通过它彼此因果。这是与战斗系统平级、需要独立打磨的核心系统。

---

## 一、核心理念：选择塑造世界

剧情系统的本质不是"可跳转的节点图"，而是"**一个被玩家选择不断改写的江湖**"。

玩家的一次选择，影响沿四个层次传播（这是本系统的脊柱）：

| 影响层次 | 例子 | 落到哪 |
|---|---|---|
| ① 自身收益/属性 | 得银两、习武功、声望涨 | `player`（属性/道具/武功） |
| ② NPC 好感与关系 | 丘处机好感+15 | `player.relations`（已存在） |
| ③ NPC 命运/阵营 | 招募段誉入队、杨康黑化、全真教敌视 | `world.npcs` / `world.factions` |
| ④ 后续剧情/其他事件 | 侠义够高→洪七公主动传功；未救人→日后被冷遇 | `condition` 查询 + `world.arcs` + 涌现事件 |

四层统一为同一件事：**对世界状态的声明式写入（consequence）与查询（condition）**。玩家不必记得自己做过什么，但**世界记得**。

## 二、架构分层（依赖方向，单向不可逆）

```
data/story/            ← 剧情内容（纯数据，只有"发生了什么"）
  schema.ts              数据模型类型
  volumes/shendiao.ts    射雕卷（纯数据节点）
  volumes/common.ts      通用奇遇
  world-events.ts        涌现事件（由世界状态触发）
  index.ts               聚合 + 查询

game/story/             ← 剧情引擎（只有"怎么发生"，不认识任何具体剧情）
  state.ts               世界状态 WorldState 的定义与初始化
  consequences.ts        声明式后果解释器（applyConsequences）
  conditions.ts          条件解释器（checkCondition）
  engine.ts              节点结算 / 流转 / 战斗衔接
  world-scheduler.ts     涌现事件扫描

screens/EventScreen.tsx ← 纯展示，调引擎
App.tsx                 ← 只做路由编排
```

**铁律**：`game/story/*` 永远不 import `data/story/volumes/*`，不认识任何剧情 id / NPC id。引擎只认 `schema.ts` 的类型。

## 三、世界状态 WorldState（核心）

### 3.1 与 Player 的关系

`Player` 保持现有结构（战斗属性 + `reputation`/`alignment`/`relations`/`gold`）。新增 `WorldState` 承载"江湖记忆"，挂在 `player.world` 上随存档持久化（旧存档迁移补默认空状态）。

- `reputation` / `relations` / `gold` 已在 `player` 上，**就地复用**，不搬迁（避免大改）。
- `world.npcs` / `world.factions` / `world.arcs` / `world.flags` 是**新增**的剧情世界维度。
- 引擎统一寻址——**作者只写"改什么"，引擎知道存哪**，不必关心字段在 player 还是 world。

### 3.2 善恶值 karma（新增数值源头）

金庸群侠传需要数值化的善恶路线，而当前 `player.alignment` 只是 `"正"|"邪"|"中"` 三值枚举。引入：

```ts
player.karma: number   // -100(邪) ~ +100(正)，0 为中性
```

`alignment` 枚举**由 karma 派生**（karma ≥ 30 → "正"；≤ -30 → "邪"；否则 "中"），保留枚举接口供 UI/旧逻辑使用。这样"善恶"既能细粒度数值演化，又能粗粒度归类。

> 决策点（待你拍板）：是让 alignment 完全由 karma 派生（只留 karma），还是两者并存（karma 驱动、alignment 可被剧情强制覆写）？我建议前者更干净。

### 3.3 WorldState 结构

```ts
interface WorldState {
  // NPC 命运：好感之外的"关系状态"（好感值仍用 player.relations，不重复存）
  npcs: Record<string, {
    alive: boolean                 // 生死（守门：已死 NPC 不再出现）
    recruited: boolean             // 是否已加入主角队伍（经典招募机制）
    faction: string                // 当前所属阵营（可随剧情转变，如杨康叛全真投金）
    fateTags: string[]             // 命运标记（如 "杨康已黑化"、"段誉已结拜"）
  }>

  // 阵营/门派：对玩家的态度 + 势力消长
  factions: Record<string, {
    attitude: number               // 对玩家的态度 -100~100
    power: number                  // 势力强弱（可用于世界事件/结局）
  }>

  // 主线/事件链进度
  arcs: Record<string, {
    beats: Record<string, "won" | "lost" | "skipped" | "done">
  }>

  flags: Record<string, boolean | number | string>  // 琐碎标记兜底

  triggeredEvents: string[]        // 已触发过的涌现事件 id（防重复）
}
```

> 字段**渐进生长**：`npcs`/`factions`/`arcs` 都是 `Record`，写新剧情时随用随加 key，无需全局重构。

## 四、因果机制：Consequence（声明式写入）

一次选择/一个战斗结局对世界产生的影响，全部声明式描述。引擎的 `applyConsequences` 认识以下种类：

```ts
type Consequence =
  // ① 自身收益/属性
  | { kind: "reputation"; delta: number }
  | { kind: "karma"; delta: number }            // 善恶值
  | { kind: "gold"; delta: number }
  | { kind: "exp"; delta: number }
  | { kind: "hp" | "mp"; delta: number }
  | { kind: "aptitude" | "attack" | "speed"; delta: number }
  | { kind: "item"; id: string; count?: number }   // 默认 1，负数=扣除
  | { kind: "skill"; id: string }                  // 习得武功（去重）
  // ② NPC 好感（写入 player.relations）
  | { kind: "relation"; npcId: string; delta: number }
  // ③ NPC 命运/阵营
  | { kind: "npcAlive"; npcId: string; alive: boolean }
  | { kind: "npcRecruit"; npcId: string; recruited: boolean }
  | { kind: "npcFaction"; npcId: string; faction: string }
  | { kind: "npcTag"; npcId: string; tag: string; add?: boolean }   // 默认 add=true
  | { kind: "factionAttitude"; factionId: string; delta: number }
  | { kind: "factionPower"; factionId: string; delta: number }
  // ④ 主线进度 + 标记
  | { kind: "arcBeat"; arcId: string; beat: string; result: "won" | "lost" | "skipped" | "done" }
  | { kind: "flag"; name: string; value: boolean | number | string }
```

> **边界**：Consequence 种类是有限枚举。遇到新需求，优先用 `flag`/`npcTag` + condition 表达；只有高频通用的变更才升级为新 kind。不为单一剧情加专属 consequence。

## 五、查询机制：Condition（声明式读取/分支）

节点、选项、事件、涌现触发都可挂条件，控制可见/可选/可触发：

```ts
type Condition =
  | { kind: "reputation"; gte?: number; lte?: number }
  | { kind: "karma"; gte?: number; lte?: number }              // 善恶区间（正线/邪线分支）
  | { kind: "relation"; npcId: string; gte?: number }          // NPC 好感门槛
  | { kind: "npcAlive"; npcId: string; alive?: boolean }
  | { kind: "npcRecruited"; npcId: string }
  | { kind: "npcHasTag"; npcId: string; tag: string }
  | { kind: "factionAttitude"; factionId: string; gte?: number }
  | { kind: "arcBeat"; arcId: string; beat: string; result?: "won" | "lost" | "skipped" | "done" }
  | { kind: "flag"; name: string; eq?: boolean | number | string }
  | { kind: "hasItem"; id: string }
  | { kind: "hasSkill"; id: string }
  | { kind: "and"; items: Condition[] }
  | { kind: "or"; items: Condition[] }
  | { kind: "not"; item: Condition }
```

## 六、涌现事件（"其他事件"被世界变化触发）

除玩家主动触发的剧情外，有一类**世界事件**：当世界状态满足条件时自动浮现，体现"世界记得你的选择"。

```ts
interface WorldEvent {
  id: string
  trigger: Condition           // 满足则可触发（如 karma ≥ 30 且未触发过）
  once: boolean                // 通常 true，触发后记入 triggeredEvents
  event: StoryEvent            // 触发后进入的剧情事件
}
```

示例——侠义之名远播，洪七公主动现身：
```ts
{
  id: "hongqi-gong-seeks-you",
  trigger: { kind: "and", items: [
    { kind: "karma", gte: 30 },
    { kind: "relation", npcId: "hongqi", gte: 0 },   // 且与洪七公无恶感
  ]},
  once: true,
  event: { /* 洪七公传降龙十八掌的剧情事件 */ },
}
```

**触发时机**（MVP）：每次回到主菜单时，`world-scheduler` 扫描 `WorldEvent`，触发第一个满足 `trigger` 且未在 `triggeredEvents` 中的，弹给玩家。未来可扩展为按天数/地点触发。

## 七、节点 / 选项 / 事件 / 流转

```ts
type Transition =
  | { type: "end" }
  | { type: "goto"; nodeId: string }                                   // 同事件下一节点
  | { type: "battle"; enemyId?: string; useLocationPool?: boolean;
      onWin?: Outcome; onLose?: Outcome; onFlee?: Outcome }
  | { type: "gotoEvent"; eventId: string }                             // 跨事件串联

interface Outcome {
  text: string
  consequences?: Consequence[]
  then?: Transition                     // 战后继续流转，默认 end（"打赢接下一段"靠它）
}

interface Choice {
  id: string
  text: string
  description?: string
  condition?: Condition                 // 不满足则隐藏
  consequences?: Consequence[]          // 选了立即结算（影响玩家+世界）
  transition: Transition
  resultText?: string
}

interface StoryNode {
  id: string
  title?: string
  text: string
  speaker?: string
  onEnter?: Consequence[]               // 进入节点时结算（如自动给奖励）
  choices: Choice[]
}

interface StoryEvent {
  id: string
  entryNode: string
  nodes: Record<string, StoryNode>
  locationId?: string
  weight?: number
  condition?: Condition
}
```

## 八、因果链示例（"选择有重量"如何落地）

> 牛家村救丘处机 → 一路影响传播

```ts
// 牛家村·力战胜利
onWin: {
  text: "丘处机长揖：\"阁下仗义出手，丘某佩服。\"传你两式全真剑法要诀，赠碎银。",
  consequences: [
    { kind: "karma", delta: 5 },                          // 侠义↑
    { kind: "reputation", delta: 5 },                     // 声望↑
    { kind: "relation", npcId: "qiuchuji", delta: 15 },   // 丘处机好感↑
    { kind: "factionAttitude", factionId: "quanzhen", delta: 10 }, // 全真教态度↑
    { kind: "item", id: "small-hp-pill", count: 2 },
    { kind: "arcBeat", arcId: "shendiao", beat: "niujia", result: "won" },
    { kind: "flag", name: "niujiacun_helped", value: true },
  ],
  then: { type: "end" },
}
```

这些写入会在**后续任何剧情**里被 condition 读到，形成因果链：
- 路过重阳宫：`condition: { kind: "factionAttitude", factionId: "quanzhen", gte: 10 }` → 解锁"求学全真剑法"隐藏节点。
- 某世界事件：`trigger: { kind: "relation", npcId: "qiuchuji", gte: 50 }` → 丘处机日后主动来寻、传更高深武功。
- 邪线分支：若牛家村选择袖手（karma 不涨/反降），后期某些正派 NPC 态度冷淡，选项被隐藏。

**这就是"每个节点都可能影响其他节点"——通过共享世界状态，无需节点间互相硬编码引用。**

## 九、引擎接口

```ts
// game/story/state.ts
createWorld(): WorldState                              // 初始空世界
deriveAlignment(karma: number): Alignment              // karma → 正/邪/中

// game/story/consequences.ts
applyConsequences(player: Player, world: WorldState, cs?: Consequence[]): { player: Player; world: WorldState }

// game/story/conditions.ts
checkCondition(player: Player, world: WorldState, cond?: Condition): boolean

// game/story/engine.ts
enterNode(player, world, event, nodeId): { node, player, world }      // 进节点（含 onEnter、可见选项过滤）
resolveChoice(player, world, event, nodeId, choiceId): { player, world, resultText?, transition }

// game/story/world-scheduler.ts
pollWorldEvent(player, world): { player, world, event?: StoryEvent }  // 扫描涌现事件
```

`App.tsx` 只按 `transition.type` 路由（end/goto/battle/gotoEvent），并在回主菜单时调 `pollWorldEvent`。**不含任何剧情业务逻辑**。

## 十、一致性与可演进性

- **生死守门**：所有让某 NPC 出场的节点，挂 `condition: { kind: "npcAlive", npcId, alive: true }`，避免"已死之人复现"。
- **去重**：涌现事件用 `triggeredEvents` 防重复；一次性剧情用 `arcBeat`/`flag` 标记已完成。
- **渐进生长**：`npcs`/`factions`/`arcs` 随写随加 key，`WorldState` 结构稳定、字段渐增，不必预先定义全部 NPC。
- **可审视**：可随时 dump `{ karma, reputation, relations, world }` 一眼看清"当前江湖状态"，便于调试与剧情校验。

## 十一、迁移路线（每阶段可跑可玩）

> 关键调整：**世界状态的 schema 骨架在阶段 1 就立起来**，即使字段先少。不能等剧情铺开再补，否则 flag 会失控。

**阶段 1｜骨架 + 数据化**
1. 定义 `schema.ts`（WorldState/Consequence/Condition/Transition/Choice/Node/Event）
2. 实现 `state.ts`/`consequences.ts`/`conditions.ts`/`engine.ts`
3. `Player` 加 `karma`、`world`；旧存档迁移（补默认空世界、karma 由原 alignment 推算）
4. 现有 `STORY_EVENTS` 闭包 → 声明式 consequences；战后收尾由 `onWin` 等驱动，移除 `afterBattle` 补丁
5. **验收**：现有剧情行为不变；WorldState 骨架就位，可 dump 查看

**阶段 2｜节点串联（直接因果）**
- 落地 `goto` + 节点表；App 维护当前节点；prerequisite 用 condition 表达
- 单事件升级为多节点连续剧情
- **验收**：可做"打赢后自动进入下一段"

**阶段 3｜世界状态 + 涌现事件（间接因果，"整体剧情"成型）**
- `npcs`/`factions`/`arcs` 读写真正生效；`karma` 驱动正邪分支
- `world-scheduler` 落地，世界事件涌现
- **验收**：选择能跨事件传播、世界记得玩家、善恶路线分化

**阶段 4｜主线 Arc / 名场面事件链**
- 聚贤庄、华山论剑、光明顶等大事件链（世界观"远期深化"目标）
- 队友招募系统接入（`npcRecruited` 落地为实际队友）

**阶段 5｜（可选）外部格式 + 编辑器**
- 剧情数据迁 JSON/YAML；评估可视化编辑器与因果依赖图

## 十二、设计原则与边界

- **内容零逻辑**：`data/story` 下不出现函数，只描述数据与 consequence。
- **引擎零剧情**：`game/story` 下不出现任何具体剧情/NPC/阵营 id。
- **有限枚举**：Consequence/Condition 种类有界；特殊性用 flag/npcTag + condition 表达。
- **世界状态前置**：schema 骨架先于大量剧情定型；字段渐进生长。
- **不过度设计**：阶段 1–3 即达成"独立、可扩展、选择有重量"的核心价值；DSL、全功能世界模拟、编辑器等留待剧情体量真正需要时。
