# 金庸群侠传 - 学习笔记

这份文档记录开发过程中学到的每一个技术概念，方便日后回顾。

---

## 第一部分：环境准备

### 1. Node.js 是什么

JavaScript 原本只能在浏览器里运行（用来做网页交互）。**Node.js 把 JavaScript 搬到了电脑操作系统上**，让 JS 可以读写文件、启动服务、管理项目。

- 在我们这个项目里，Node.js 是整个技术栈的"地基"——跑 Vite 开发服务器、执行 npm 命令，都靠它。
- 我们安装的是 LTS（长期支持版）v24.18.0，LTS 版本更稳定，适合开发用。
- 安装位置：`C:\Program Files\nodejs`。Node.js 本体是系统级组件，留在 C 盘是标准做法，体积不大。

**类比**：如果 JavaScript 是"乐高积木"，Node.js 是"工作台"，npm 是"乐高商店"。

### 2. npm（包管理器）

`npm` 是随 Node.js 一起安装的**包管理器**，相当于"第三方代码的应用商店"。

- 世界上有无数开发者写好了现成的功能包（比如 React、工具库），上传到 npm 仓库。
- 你用 `npm install 包名` 就能把别人写好的代码下载到自己项目里，不用重复造轮子。
- 我们的 React、TypeScript 等都是通过 npm 装进来的。

### 3. npm 缓存

npm 每次下载包时，会先存一份到本地**缓存目录**。

- **作用**：多个项目用同一个包时，不用重复下载，直接从缓存复制，省时省流量。
- **为什么改到 D 盘**：npm 默认把缓存堆在 C 盘的 AppData 下，项目一多就占空间。我们改到了 `D:\dev\npm-cache`，让 C 盘只保留系统组件。
- **注意区分**：缓存是"下载仓库"（D:\dev\npm-cache），项目里的 node_modules 才是项目真正在用的依赖。npm 是从缓存把文件复制到各项目的 node_modules。

**类比**：缓存是"下载工具的下载文件夹"，node_modules 是"每个项目实际用的零件盒"。

### 4. PowerShell 执行策略

安装时遇到过一个错误：`running scripts is disabled on this system`。

- Windows 出于安全考虑，默认禁止运行 PowerShell 脚本（.ps1 文件）。
- npm 的某些命令是以 .ps1 脚本形式提供的，所以被拦住了。
- 我们用 `Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser` 放开了限制：本地脚本可运行，从网上下载的脚本必须有数字签名。这是开发环境的常规配置，安全可控。

## 第二部分：技术栈与项目结构

### 1. Vite（构建工具 / 开发服务器）

Vite 是现代前端的**构建工具**，负责两件大事：

- **开发时**：启动一个本地开发服务器（dev server），你改代码它会自动刷新页面（这叫 HMR，热模块替换），写起来特别顺手。
- **发布时**：把整个项目"打包"成浏览器能直接运行的静态文件（HTML/CSS/JS），方便部署上线。

**为什么选 Vite**：速度快（基于浏览器原生 ES 模块，启动几乎秒开）、配置简单、是 2024 年起的前端默认选择，取代了老牌的 Webpack。

**踩过的坑**：Vite 处理 CSS 时会读 package.json 当配置。如果 package.json 开头带了 BOM（字节顺序标记），解析就会失败，导致 CSS 加载报 500 错误，整个页面渲染不出来。解决方法是确保所有文件用 UTF-8 无 BOM 编码。

### 2. React（UI 框架）

React 是 Facebook 出品的**前端框架**，核心思想是"组件化"：

- 把界面拆成一个个独立的"组件"（Component），每个组件负责自己的一小块界面。
- 组件用 JSX 语法写（看起来像 HTML 写在 JS 里），直观易读。
- 数据变了，React 自动帮你更新界面，不用手动操作 DOM。

我们项目里的组件：
- `App.tsx` 是根组件，管理标题界面 / 主界面之间的切换。
- `TitleScreen.tsx` 是标题界面组件。
- `MainScreen.tsx` 是角色面板组件。

**useState**：React 的一个核心钩子（hook），让组件能"记住"数据。比如标题界面要记住"当前是否在输入名字"这个状态。

### 3. TypeScript

TypeScript 是 JavaScript 的**增强版**，最大的区别是加了**类型系统**：

- 普通 JS 里，变量可以是任何类型，写错了也不报错，运行时才崩。
- TS 里，你要声明变量的类型（比如 `name: string`），写错了编辑器立刻红线提示，编译时直接报错。

**interface（接口）**：用来定义一个数据的"形状"。我们在 `types/index.ts` 里定义了 Skill（武功）、Player（角色）等接口，所有地方用到的数据都得符合这些形状。这就像给数据画了张图纸，谁用都得照图纸来。

类型系统的好处：项目越大、数据越多，它越能帮你挡住低级 bug。对这种数据密集型的游戏几乎是必需。

### 4. 数据驱动设计

这是整个项目架构的核心理念：**把"数据"和"逻辑"分开**。

- 数据（data 文件夹）：武功、剧情事件、门派……都是纯数据，比如 skills.ts 里就是一个武功数组。
- 逻辑（game 文件夹）：战斗计算、存档读写……是处理数据的代码。

好处：以后想加一本新书、一个新武功，只要往 data 文件夹加一条数据，不用动核心代码。这就是"长期可迭代"的关键。

### 5. 项目目录结构

```
jinyong/
├── index.html          # 网页入口
├── package.json        # 项目配置和依赖清单
├── vite.config.ts      # Vite 配置
├── 学习笔记.md          # 本文档
└── src/                # 源代码
    ├── main.tsx        # JS 入口，挂载 React 到页面
    ├── App.tsx         # 根组件，管理界面切换
    ├── App.css         # 全局样式
    ├── types/          # 类型定义（数据的形状）
    ├── data/           # 游戏数据（武功等）
    ├── game/           # 核心逻辑（角色、存档）
    ├── screens/        # 游戏界面（标题、主城）
    ├── components/     # 可复用 UI 零件（预留）
    └── utils/          # 工具函数（预留）
```

### 6. 存档系统（localStorage）

我们的存档方案分两层：

- **基础存档**：用浏览器的 `localStorage`，存档永久保存在本地浏览器，关掉再开、明天再玩都在。限制是存档绑定在"浏览器+设备+域名"上，换设备读不到。
- **存档码导入导出**：把存档编码成 Base64 文本码，玩家可以复制下来，换设备时粘贴导入。零后端解决跨设备问题。

### 7. 文件编码：UTF-8 与 BOM

开发中踩了一个坑：PowerShell 的 Set-Content 默认会给文件加 BOM（Byte Order Mark，字节顺序标记），导致 JSON 解析失败。

- **BOM 是什么**：文件开头加的 3 个特殊字节（EF BB BF），本来用来告诉程序"这是 UTF-8 编码"。
- **为什么会有问题**：很多解析器（包括 JSON.parse）不认 BOM，把它当成正文内容，导致解析报错。
- **经验教训**：写代码文件一律用 **UTF-8 无 BOM** 编码。VS Code 默认就是无 BOM 的，可以放心用。

## 第三部分：战斗系统

### 1. 战斗流程设计

回合制战斗，状态机驱动。每个回合分四个阶段（phase）：

- `select`：玩家选招式
- `enemyTurn`：敌人自动出招（有 0.8 秒延迟，营造节奏感）
- `won`：敌人倒下，结算奖励
- `lost`：玩家倒下，败北

phase 的变化由 React 的 useState 管理，useEffect 监听 phase 变化来触发敌人回合。这种"状态机"模式让复杂的回合逻辑变得清晰可控。

### 2. 伤害公式

```
基础伤害 = 武功威力 + 攻击力 - 防御力
实际伤害 = 基础伤害 × (1 + 随机浮动 ±15%)  ; 暴击时再 ×1.5
```

还加入了**闪避判定**：双方身法差每 10 点约提高 12% 闪避率，上限 35%。身法高的角色更容易躲开攻击，这让"轻功"类属性有了实战意义。

### 3. 副本机制（重要）

战斗中用的是角色的**副本**（structuredClone），不是原始存档。原因：

- 战斗中血量会变，但不能直接改存档——万一战斗中途退出，存档就被污染了。
- 用副本打，只有战斗胜利结算后，才把结果写回存档。
- 这是"不可变数据"思想的体现：核心数据只读，操作时用副本，最后再合并。

### 4. useEffect 与副作用

React 组件里，`useEffect` 用来处理"副作用"——不是直接渲染界面、而是在某个状态变化时触发的外部操作。

战斗里我们用它来：
- 日志自动滚动到底部（log 变化时触发）
- 敌人回合的自动执行（phase 变成 enemyTurn 时触发，带延迟）

依赖数组（第二个参数）控制"什么时候触发"：只写 `[phase]` 表示只在 phase 变化时执行，避免重复触发。

### 5. 数据与逻辑分离（再次体现）

战斗系统也遵循数据驱动：
- `data/enemies.ts`：纯数据，三个敌人的属性。加敌人就是加一条。
- `game/battle.ts`：纯逻辑，伤害计算、AI、结算。不知道具体敌人是谁。
- `screens/BattleScreen.tsx`：纯界面，只负责展示和接收点击，计算全交给 battle.ts。

这种三层分离的好处：改伤害公式只动 battle.ts，加新敌人只动 enemies.ts，改界面只动 BattleScreen，互不干扰。

## 第四部分：门派系统

### 1. 解锁高级武功的意义

门派系统的核心价值：让玩家从"只会长拳"变成"会降龙十八掌"。绝学数据本来就在 skills.ts 里备好了，门派系统给它们一个"入口"——拜师才能学。

- 免费武功：每个门派都有一门免费入门武功（如少林九阳、武当太极），1级就能学。
- 进阶武功：需要达到等级要求 + 花银两（如降龙十八掌要 6 级 500 两），制造成长目标。

### 2. 条件判断与禁用态

UI 里的"等级不足/银两不足"按钮是禁用态（disabled），这是前端常见模式：

- 数据驱动：根据玩家等级和银两算出能不能学。
- 不能学时按钮置灰、不可点，同时文字说明原因（等级不足/银两不足）。
- 这样玩家一眼就知道"还差什么"，而不是点了没反应一脸懵。

### 3. 组件复用

注意一个设计：门派界面和主界面都有"武功列表"，但它们是各自独立的组件，没有强行复用。原因是：

- 主界面的武功列表是"展示已学武功"，只读。
- 门派界面的武功列表是"选择学习"，带按钮和状态。
- 两者外观和交互都不同，强行抽公共组件反而增加复杂度。

这是"避免过度抽象"的体现：只有当两处真的高度相似、且会重复多次时，才值得抽组件。

### 4. 时间的推进

游戏里每个行动（修炼、闯荡、拜师）都会推进"天数"。这是一个简单但有效的游戏节奏设计：

- 天数是隐形的压力，让玩家有"时间在流逝"的感觉。
- 未来可以加"限时不等人"的剧情（比如某事件第 30 天才触发）。
- 目前只是计数，但架构上预留了扩展空间。

## 第五部分：战斗动画（CSS 动画 + 状态控制）

### 1. 动画的本质：状态 + 过渡

战斗动画没有用任何动画库，纯靠 **CSS 动画 + React 状态切换**实现。核心套路是：

- 某件事发生时（比如"玩家出招"），设置一个临时状态（比如 skillFlash = "降龙十八掌"）。
- 这个状态触发对应的 CSS 动画类（比如 .skill-flash）。
- 动画播完后，用 setTimeout 把状态清掉。

这就是"声明式动画"：你只管"什么时候开始"，动画的细节交给 CSS。比写一堆 if/else 手动操控 DOM 简洁得多。

### 2. CSS @keyframes 关键帧

CSS 动画的核心是 @keyframes，它定义动画的"每一帧长什么样"。比如招式飞屏：

```css
@keyframes skillFlashIn {
  0%   { opacity: 0; transform: scale(0.3); }   /* 起始：小且透明 */
  25%  { opacity: 1; transform: scale(1.2); }    /* 放大弹出 */
  60%  { opacity: 1; transform: scale(1); }      /* 回到正常 */
  100% { opacity: 0; transform: scale(1.1); }    /* 淡出 */
}
```

百分比表示"动画进度的几分之几"。浏览器会自动在关键帧之间插值（补间），所以你只要画几个关键画面，中间的过渡它帮你算。

### 3. 为什么用 setTimeout 控制时长

React 的临时状态（skillFlash、floats、shake）需要在动画播完后清除，否则状态一直留着。做法：

- CSS 动画本身有固定时长（比如 0.7s）。
- 用 setTimeout 在对应毫秒后把状态设回 null/空数组。
- 这样"动画播完"和"状态清除"的时间是同步的。

这是个简单可靠的套路，不需要复杂的动画库。真正的复杂动画（比如精灵图逐帧、物理引擎）才需要专门库。

### 4. 动画与游戏逻辑的解耦

注意一个设计：动画是"展示层"，不参与伤害计算。战斗逻辑（battle.ts）该算多少伤害还是算多少，动画只是把这些数字"演"出来。这样改动画不会影响游戏平衡，改数值也不会破坏动画——两层完全独立。

## 第六部分：四类武功体系 + 状态系统

### 1. 武功分类的意义

把武功从"都是攻击"重构为四类，每类有完全不同的战斗作用：

- **外功**：直接造成伤害（降龙十八掌、六脉神剑）
- **内功**：给自己挂增益/治疗状态（九阳神功回春、易筋经加防）
- **轻功**：提升闪避与身法（凌波微步、梯云纵）
- **奇门**：给敌人挂毒/削弱/眩晕（千蛛万毒手、摄魂大法）

这让战斗从"比谁伤害高"变成策略组合："先开九阳增益 → 用毒削弱 → 外功爆发"。

### 2. 状态效果系统（StatusEffect）

战斗单位身上可以挂状态，每回合自动结算：

- **中毒**：每回合开始掉血
- **回春**：每回合开始回血
- **增益**（攻/防/身法）：在 effectiveStats() 里临时加成
- **眩晕**：跳过一回合
- **护盾**：减免伤害，被击后扣减

状态有 duration（持续回合）和 potency（强度），每回合 duration-1，归零后消失。这是回合制策略游戏的核心机制。

### 3. React 的闭包陷阱与 ref 解法

这次踩了一个经典坑：用 setTimeout 在异步回调里读 state，读到的是闭包创建时的旧值，导致状态结算后流程卡住。

解决方法是用 **useRef** 持有最新状态：

```tsx
const pRef = useRef(p)
pRef.current = p  // 每次渲染都同步

// 异步回调里读 pRef.current，永远是最新值
```

**为什么需要 ref**：React 的 setState 是异步的，而且在 setTimeout 的闭包里，变量是被"冻结"在闭包创建那一刻的值。ref 是个可变容器，current 属性随时可读写，不受闭包限制。这是 React 里处理"异步逻辑需要最新状态"的标准套路。

### 4. 数据兼容性设计

旧的存档（没有 statuses 字段）在加载时做了兼容：

```tsx
if (!p.statuses) p.statuses = []  // 旧存档补上空数组
```

游戏迭代时，数据结构会变。给加载逻辑加兼容处理，能避免老玩家存档报废——这是长期迭代必须养成的习惯。
